## 2. システム概要

### 2.1 システム名
AI面接コーチ

### 2.2 システムの目的
面接が苦手な求職者に対し、AIとの模擬面接、職務経歴書に基づく質問生成、練習履歴の蓄積、振り返りフィードバック、追加課金による継続利用機能を提供し、面接力向上を支援する。

### 2.3 背景
- 面接練習は一人では継続しづらく、客観的なフィードバックも得にくい
- 自身の職務経歴に沿った質問で練習したいニーズがある
- 振り返りを蓄積し、改善を継続できる仕組みが必要である
- サービス運用コストは可能な限り抑える必要がある
- AI基盤は Azure 経由で利用する方針である

### 2.4 対象ユーザー
- 面接練習を行う一般ユーザー
- 管理、監視、問い合わせ対応を行う運営担当者

### 2.5 提供価値
- AIとの対話による模擬面接練習
- 職務経歴書ベースの質問生成
- 練習履歴と振り返りによる継続学習
- 利用時間に応じたクレジット管理と追加課金

## 3. 対象範囲

### 3.1 今回の実装範囲
- AIとの対話機能
- React / TypeScript / Tailwind CSS によるフロントエンド MVP
- Django による JSON API バックエンド MVP
- 開発用デモ認証
- 練習履歴の保存、表示、削除
- 職務経歴書アップロード
- 練習メッセージ管理
- 振り返り生成
- 30分単位のクレジット管理
- Stripe Checkout Sessions 連携用エンドポイント
- Azure OpenAI 連携アダプタ
- Azure OpenAI 未設定時のローカル応答

### 3.2 今後実装予定の要素
- Cognito による本番認証
- パスキー認証
- 電話番号認証と重複登録防止
- パスワードリセット機能
- Cloudflare Workers を活用した配信最適化
- 既存システムからの Cognito ユーザーデータ移行

### 3.3 対象外
- 企業向け採用管理機能
- 有人面接官マッチング機能
- 現行システム全体の全面刷新

## 5. 非機能要件

### 5.1 性能
- 主要画面は通常操作で 3 秒以内の応答を目標とする
- AI対話開始までの待ち時間は最小化する
- 履歴表示はユーザー体感上ストレスのない速度を確保する
- Cloud Run のコールドスタートによる初回遅延を考慮し、最小インスタンス数は 0 を前提として初期化処理の軽量化、ローディング表示、再試行導線を整備する

### 5.2 可用性
- AI連携障害時はローカル応答による代替を可能とする
- 認証、課金障害時はユーザーに明確な状態を提示する
- 主要データは定期バックアップを行う
- Cloud Run の新規インスタンス起動中は一時的に応答遅延が発生し得るため、フロントエンドではローディング表示と再試行可能な導線を提供する

### 5.3 拡張性
- フロントエンドとバックエンドを分離する
- 認証、AI、課金はアダプタ層で疎結合化する
- 将来的な機能追加や外部サービス差し替えに対応しやすくする

### 5.4 セキュリティ
- HTTPS による通信暗号化
- JWT 検証
- 個人情報の保護
- アップロードファイル制御
- 操作ログ、監査ログ保持

### 5.5 コスト
- インフラは可能な限り安価な構成とする
- 利用可能であれば RDS を活用する
- OpenAI 利用は Azure 経由に限定する
- 職務経歴書ファイルは Amazon S3 に保存し、DB には参照用メタ情報のみ保持する
- AWS で新規に作成するリソースは AWS CDK で管理する
- データベースは既存環境を利用し、新規作成対象外とする

## 6. システム構成

### 6.1 技術構成
- フロントエンド: React, TypeScript, Tailwind CSS
- バックエンド: Django, Python3
- 配信・エッジ: Cloudflare Workers
- アプリ実行基盤: Google Cloud Run
- 認証基盤: 開発時 Demo Auth、本番時 Amazon Cognito
- AI基盤: Azure OpenAI Realtime API
- 決済基盤: Stripe Checkout Sessions
- データベース: 既存のリレーショナルデータベース
- ファイル保管: Amazon S3 を用いた職務経歴書アップロード用ストレージ
- AWS リソース管理: AWS CDK

### 6.2 全体構成図
```text
[ユーザー]
   |
[React Frontend]
   |
[Cloudflare Workers]
   |
[Django JSON API on Cloud Run]
   |------ [Auth Adapter: Demo / Cognito]
   |------ [AI Adapter: Azure OpenAI Realtime API]
   |------ [Billing Adapter: Stripe]
   |------ [RDB]
   |------ [Amazon S3]
```

### 6.3 構成方針
- UI と API を分離する
- DB への直接アクセスはバックエンドからのみ許可する
- 外部サービス連携はアダプタ層を介して行う
- MVP 段階でも本番移行を見据えた分割を行う
- 起動遅延の影響が大きい面接開始 API や認証後初回 API については Cloud Run の最小インスタンス数 0 を前提に、依存初期化の遅延読み込み、ヘルスチェック整備、フロント側待機表示を前提とする

## 9. データ設計

### 9.1 主要エンティティ
- User
- UserProfile
- ResumeFile
- InterviewSession
- InterviewMessage
- Reflection
- CreditBalance
- CreditTransaction
- PaymentSession
- AuditLog

### 9.2 エンティティ概要

#### User
- ユーザーの基本識別情報を保持する
- メールアドレス、氏名、電話番号、認証方式を管理する

#### ResumeFile
- アップロードされた職務経歴書ファイルを管理する
- ユーザーと紐付ける

#### InterviewSession
- 面接練習 1 回分のセッションを管理する
- 開始時刻、終了時刻、状態、消費時間を保持する

#### InterviewMessage
- 面接中の発話、応答メッセージを保持する
- セッションに紐付ける

#### Reflection
- セッションごとの振り返り内容を保持する
- 良かった点、改善点、助言を保存する

#### CreditTransaction
- クレジットの増減履歴を保持する
- 課金、付与、消費の監査証跡とする

### 9.3 データ設計方針
- ユーザーと履歴は 1 対多で管理する
- セッションとメッセージは 1 対多で管理する
- 課金履歴は取り消しや調査に耐えられる形で保存する
- 個人情報はログ出力時にマスキングする

## 10. 外部連携設計

### 10.1 Azure OpenAI 連携
- 面接対話生成に利用する
- 振り返り生成に利用する
- リアルタイム会話体験を提供する
- 未設定時はローカル応答で代替する

### 10.2 Cognito 連携
- Hosted UI または Amplify Auth を利用する
- フロントエンドは取得した JWT を API に送信する
- バックエンドでトークン検証を実施する

### 10.3 Stripe 連携
- Checkout Sessions により追加課金を受け付ける
- 決済成功時にクレジットを反映する
- 課金結果整合性の確認を行う
