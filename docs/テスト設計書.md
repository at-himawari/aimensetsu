# AI面接コーチ テスト設計書

## 1. 文書概要

### 1.1 目的
本書は、AI面接コーチの品質を担保するためのテスト観点、テスト種別、重点確認項目を整理するものである。特に認証、課金、AI フォールバック、Cloud Run 起動遅延の影響を重視する。

### 1.2 対象
- フロントエンド
- バックエンド API
- DB 更新処理
- 外部連携
- E2E シナリオ

## 2. テスト方針
- 単体テスト、結合テスト、E2E テストを組み合わせる
- 課金、クレジット、認証は回帰しやすいため優先的に自動化する
- 外部連携はモックと実連携確認を使い分ける
- 異常系と境界値を正常系と同等以上に重視する

## 3. テスト対象と観点

### 3.1 認証・認可
| 観点 | 内容 |
| --- | --- |
| デモ認証 | `X-Demo-User` で認証できる |
| JWT 認証 | 正常 JWT で認証できる |
| JWT 異常 | 期限切れ、不正署名、audience 不一致で `401` |
| 認可 | 他人のリソース参照を拒否する |
| 管理 API | `admin` 以外は拒否する |

### 3.2 職務経歴書
| 観点 | 内容 |
| --- | --- |
| アップロード正常 | PDF を保存できる |
| 形式異常 | 非許可形式を拒否する |
| サイズ超過 | 上限超過ファイルを拒否する |
| S3 保存失敗 | DB 不整合なくエラー終了する |
| 削除 | 論理削除後に一覧表示されない |

### 3.3 面接セッション
| 観点 | 内容 |
| --- | --- |
| 開始正常 | 残高ありで開始できる |
| 開始異常 | 残高不足で開始できない |
| 所有権 | 他人の RESUME では開始できない |
| 終了処理 | 消費時間、残高、履歴が整合する |
| 状態制御 | `completed` セッションにメッセージ追加できない |

### 3.4 AI 応答・フォールバック
| 観点 | 内容 |
| --- | --- |
| Azure 正常 | 通常応答が返る |
| Azure タイムアウト | フォールバックへ切り替わる |
| Azure 障害 | `used_fallback=true` を返す |
| 振り返り生成 | 通常生成、簡易生成の両方を確認する |
| UI 表示 | 簡易応答モードを表示する |

### 3.5 課金・クレジット
| 観点 | 内容 |
| --- | --- |
| Checkout Session 作成 | 正常に作成される |
| 冪等性 | 同一 `Idempotency-Key` で重複作成しない |
| Webhook 正常 | クレジットが一度だけ加算される |
| Webhook 重複 | 二重加算されない |
| 残高減算 | セッション終了時に正しく減算される |
| 境界値 | 残高 0 分、1 分、30 分などで想定通り動作する |

### 3.6 Cloud Run 起動遅延
| 観点 | 内容 |
| --- | --- |
| 初回遅延 | 初回 API 応答が遅い場合でも UI が待機状態を表示する |
| タイムアウト | フロント、バック双方のタイムアウト設定が適切である |
| 再試行 | 初回失敗後に再試行導線が機能する |
| 面接開始 | 起動遅延時でも UX が破綻しない |

## 4. テスト種別

### 4.1 単体テスト
- 認証アダプタ
- AI アダプタ
- 課金計算ロジック
- クレジット更新ロジック
- バリデーション

### 4.2 結合テスト
- API と DB の整合性
- API と S3 の連携
- API と Stripe Webhook 処理
- API と Azure OpenAI フォールバック切替

### 4.3 E2E テスト
- ログインから面接完了まで
- 履歴参照、削除
- 課金購入から残高反映まで
- AI フォールバック表示確認

## 5. 重点テストケース

### 5.1 認証
1. 正常 JWT で `/api/auth/me` が成功する
2. 不正 JWT で `401` になる
3. 一般ユーザーが `/api/admin/health` にアクセスすると `403` になる

### 5.2 面接
1. 残高ありで面接開始できる
2. 残高不足で課金導線へ遷移する
3. 面接終了時に `interview_sessions.status=completed` になる

### 5.3 AI フォールバック
1. Azure OpenAI 正常時は `ai_mode=azure`
2. Azure OpenAI 障害時は `ai_mode=fallback`
3. フォールバックでも履歴保存が継続される

### 5.4 課金
1. Checkout Session が作成される
2. Webhook 受信で残高が加算される
3. 同じ Webhook を再送しても二重加算されない

### 5.5 RESUME
1. PDF を S3 に保存できる
2. S3 保存失敗時に `resume_files` が不整合な状態で残らない
3. 論理削除済み RESUME が新規面接開始画面に出ない

## 6. テストデータ方針
- 一般ユーザー、管理者ユーザーを用意する
- 残高あり、残高なしのユーザーを用意する
- 通常応答用とフォールバック確認用の AI モックを用意する
- Stripe Webhook 正常イベント、重複イベント、失敗イベントを用意する
- 小サイズ PDF とサイズ超過ファイルを用意する

## 7. 実行手段

### 7.1 バックエンド
```bash
cd backend
python manage.py test
```

### 7.2 フロントエンド
```bash
cd frontend
npm run test
npm run e2e
```

- 認証切替が環境ごとに正しい
- S3 バケット設定が本番値を向いている
- Stripe 本番鍵、Webhook 設定が正しい
- Cloud Run 初回起動遅延時の UX が許容範囲である
- 課金とクレジット反映に二重実行がない
